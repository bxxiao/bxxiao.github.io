Condition

注意要进入 condition 队列需要先确保当前线程已获取到AQS的资源（已获取到锁）。

AQS 中为每个 Condition 维护了一个条件队列，线程通过 Condition.await() 进入条件队列阻塞，被唤醒后（signal）转移到同步队列中等待竞争资源。

signal()

大概总结

  • 获取到资源的线程通过 condition.await() 进入条件队列阻塞。
    • 首先进入条件队列尾
    • 释放获取的所有资源
    • 通过 park 阻塞
    • 当 node 进入同步队列或被中断则唤醒
  • 被 signal 时会将对应 node 移动到同步队列，被中断也会移动到同步队列
  • 移动到同步队列后在 acquireQueued 中竞争资源、阻塞
  • 中断也是同样是等到获取到资源后才处理
    • 如果是 signal 之前被中断，抛出异常
    • 之后，补上中断

ReentrantLock

ReentrantLock 中定义了一个 Sync 抽象静态内部类，它是 AQS 的子类，一些主要方法：

Sync 有 2 个子类，分别对应公平锁和非公平锁。

非公平锁实现

NonfairSync 类：

非公平的体现：

  • NonfairSync#lock 中，一进去就通过 compareAndSetState(0, 1) 尝试获取资源
  • nonfairTryAcquire 中，如果发现当前存在资源，同样直接使用 CAS 尝试获取资源
  • 这 2 个操作都没有判断当前同步队列释放有阻塞的线程，而是直接尝试抢占资源，没有遵循“先来先到”，即“非公平”。

公平锁实现

公平的体现：

  • 获取资源之前，会通过 hasQueuedPredecessors() 判断同步队列中是否存在线程在等待，如果存在，则放弃获取资源，进入同步队列呆着。如果队列为空，或当前线程对应的 Node 是 head，则返回 false 。

跟synchronized比较

相对于 synchronized 它具备如下特点

  • 可中断
  • 可以设置超时时间
  • 可以设置为公平锁
  • 支持多个条件变量
  • 与 synchronized 一样，都支持可重入

使用ReentrantLock的lock()方法是不可打断的，需要使用lockInterruptibly()方法，若当前线程因获取不到锁而阻塞时调用其interrupt()方法，会让其抛出InterruptedException并放弃等待。
